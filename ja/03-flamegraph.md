# フレームグラフの読み方

フレームグラフは「**プログラムがどこで時間を使ったか**」の絵です。初めてだと
読み方にコツが要るので、ここでは前提ゼロからとても丁寧に説明します。

## まず、これは何の絵か

prperf(rperf)は計測中、プログラムを**ものすごい回数パシャパシャと写真に撮り**、
「そのとき何の処理が動いていたか(コールスタック)」を記録しています。その写真を
何千枚も重ねて集計したのがフレームグラフです。

- **1 つの箱 = 1 つのメソッド**(関数)。
- **箱の幅 = そのメソッド(とその子で呼ばれた処理)に費やされた時間の割合**。
  つまり「その処理が写っていた写真の枚数」。**幅が広い = 重い。**
- **縦の積み重なり = 呼び出しの深さ**。下が入口、上に行くほど「実際に動いて
  いた末端(leaf)」。

文章より絵です。下の例を見てください(下が入口、上が末端):

```
上(末端)  │                  ┌───────────────┐
          │                  │   String#*    │  ← 実際に CPU を使っていた処理(幅広=重い)
          │        ┌─────────┴───────┬───────┘
          │        │  JSON.generate  │ …
深さ ↕    │   ┌────┴─────────────────┴────────────┐
          │   │           Integer#times           │  ← ループ(幅は広いが中身を呼んでいるだけ)
下(入口)  │   ┌───────────────────────────────────┐
          │   │              <main>               │  ← プログラム全体(常にほぼ 100% 幅)
          │   └───────────────────────────────────┘
```

ここから読み取れること:

- 一番下の `<main>` は幅 100%。これは「プログラム全体」なので**当たり前**。
  ここを見て驚かないでください。
- 本当に時間を食っているのは、**上のほうにある幅の広い箱**(この例では
  `String#*`)。ここが「実際に CPU が張り付いていた場所」です。

## いちばん大事な 2 つのルール

1. **幅だけを見る。** 幅 = コスト。「この箱の幅をゼロにできたら、それだけ速く
   なる」と考えてください。
2. **高さは気にしない。** 高い塔(深いネスト)でも幅が細ければ安い。逆に
   ぺったり低くても幅が広ければ重い。**ノッポの塔より、横に広い台地を探す。**

## やってしまいがちな勘違い

- **左右は時間の順番ではありません。** 「左から順に実行された」ではなく、
  単に名前順などで並べているだけ。タイムラインとして読まないこと。
- **単体表示の色に意味はありません。** 箱を見分けるための色分けで、「赤い=熱い」
  ではありません(後述の **diff モードだけ**色に意味があります)。
- **一番下の箱が 100% 幅でも正常です。** それは全体を表しているだけ。

## 読む手順(レシピ)

1. **上のほうの、幅の広い箱**を探す。そこが自分の時間の使いどころ。
2. その箱を**クリックして拡大**(ズーム)。その部分木だけが画面いっぱいに
   広がり、内訳が見やすくなる。
3. 気になるメソッド名があれば**検索**して、グラフ全体で光らせる。同じメソッドが
   あちこちに散っていても、合計の幅で「実は重い」と分かる。
4. 絵が苦手なら **Top タブ**(フラットな表)を見る。メソッドが self / 累積の
   重み順に並ぶので、上から潰していけばよい。

## prperf でいちばん使うのは「diff モード」

Check Run の diff リンクから開くと、viewer は **diff モード**になります。ここが
回帰調査の本番です。

- **色に意味が出ます。** base と比べて **占有率が増えたメソッドが赤、減ったのが
  青**、ほぼ変化なしが中間色。
- なので diff モードでは「**一番広い箱**」ではなく「**一番赤い箱**」を探します。
  それが「この PR で重くなった場所」です。
- 読み分け:
  - **広くて赤い** = もともと重かった処理が、さらに悪化した。最優先。
  - **細いけど真っ赤** = 新しく現れた/割合が急増した処理。新規の犯人かも。
  - 幅は head の占有率、色は base からの増減、と覚えてください。

## viewer の便利操作

- **クリック** … その箱にズーム(部分木を全幅表示)。
- **検索ボックス** … 名前で絞り込み・ハイライト。`JSON` などで関連箇所を一括把握。
- **Top / Tags タブ** … 表で見たい人向け。self / 累積の重み順。
- **Shift+クリック** … メソッドを pin。スナップショット間での占有率の推移を
  スパークラインで表示(時間旅行)。
- **j / k** … 新しい/古いスナップショットへ移動。

## 補足: この絵は「時間」の絵

prperf のフレームグラフは**時間(CPU の重み)**の絵です。アロケーション数や GC
回数は **summary の数字**(Check Run の表)で見ます。「alloc が増えた」回帰では、
フレームグラフで**時間が増えた(赤くなった)メソッド**を手がかりに、「そこで
余計にオブジェクトを作っていないか」を当たる、という使い方になります。

## まとめ(これだけ覚えれば読める)

- 幅 = コスト、高さは無視。
- 上の幅広 = 自分の時間の使いどころ。
- 左右は順番ではない。単体の色に意味はない。
- prperf の diff は「**赤い箱を探す**」。それが今回悪くなった場所。
